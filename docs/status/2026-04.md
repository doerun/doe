### #4 â€” 10% phase-asymmetry threshold replaced with all-zero-vs-any-material formulation

CLAUDE.md non-negotiable #11: "if one side reports zero in a timing phase while the other reports material cost ... the timing scopes are measuring different things." The existing median-fraction threshold (`_PHASE_ASYMMETRY_THRESHOLD = 0.10`) only fires when the non-zero side's median crosses 10% of total â€” permitting a steady-state-zero side paired with intermittent spikes whose median is 8% to pass.

New formulation in `bench/native_compare_modules/comparability.py`:

- `_PHASE_ZERO_EPSILON: float = 1e-9` â€” samples below this are treated as zero (float-noise tolerance)
- `_PHASE_MATERIAL_FLOOR_FRACTION: float = 0.005` â€” a sample is "material" if phase fraction â‰Ą 0.5% of `executionTotalNs` (20Ă— stricter than the old 10% median)
- `_PHASE_MATERIAL_MIN_SAMPLES: int = 2` â€” the non-zero side must have at least 2 material samples, so a single spike doesn't false-fire
- `_all_samples_zero(values)` / `_material_sample_count(values)` helpers

Gate logic in `comparability_runtime.py::assess_timing_phase_equivalence`, `assess_submit_scope_equivalence`, and `claimability.py::assess_phase_equivalence`:

> if one side is all-zero AND the other side has â‰Ą `_PHASE_MATERIAL_MIN_SAMPLES` samples â‰Ą `_PHASE_MATERIAL_FLOOR_FRACTION`, the phases are measuring different things and the comparison is not claimable.

The legacy `_PHASE_ASYMMETRY_THRESHOLD = 0.10` is retained in the details payload for backwards-compatible diagnostic reporting but no longer controls the gate. The new `phaseGateFormulation: "all-zero-one-side-vs-any-material-other-side"` field on the details payload documents the active rule.

Intentional behavior: near-zero legitimate signals (warm-cached compute setup_ns that medians to ~0.1% of total) no longer false-fire; a steady-state-spike profile with 2+ samples above 0.5% still correctly fires even when the median is below 10%.

## Prior audit fixes landed: dispatch-count parity fail-closed; release-path strict-mode guard; breadth-config `allowBaselineNoExecution` rationale pinned (2026-04-16 UTC)

Three of the four items queued from the earlier fairness audit (see `Next-session follow-up queue` in the 2026-04-16 pipeline-cache entry below) are now in code. The remaining two items (Zig-side trace_meta emission for pipeline-cache warmup telemetry; `--no-pipeline-cache` flag on `doe-zig-runtime`) require Mac hardware and stay queued.

### What landed

**1. Dispatch-count parity fail-closed on unknown counts.**
`bench/native_compare_modules/compare_assessment.py::compare_execution_shapes` previously short-circuited to `(True, "")` when either side's `executionDispatchCount` set contained no `>= 0` value â€” i.e. an unknown-dispatch comparison passed silently. The check now fails with `dispatch counts unknown for row/success=â€¦` when either side has no known counts, independent of the other side's state. The prior "differ" branch is preserved. Verified against `bench/out/apple-metal/compare/20260409T210659Z/` artifacts that none of the recent compare runs legitimately report unknown dispatches, so the tightening is net-no-op for existing valid artifacts but closes the silent loophole going forward.

**2. Release-path strict-mode guard.**
`bench/lib/compare_claim_artifacts.py` now exports `RELEASE_REQUIRED_COMPARABILITY_MODES = ("strict",)`, `comparability_mode()`, and `ensure_release_strict_comparability()`. The enforcer is invoked from:

- `bench/runners/run_blocking_gates.py::main` immediately after `report_path.exists()` â€” this covers `run_release_pipeline.py`, `run_market_readiness_bundle.py`, and `run_release_claim_windows.py` transitively (those paths invoke `run_blocking_gates.py`).
- `bench/runners/publish_apple_runtime_release.py::ensure_compare_step_acceptable` on both the `step["pass"] == True` and `returnCode == 3` branches â€” `publish_apple_runtime_release.py` runs its own gates and does not transit `run_blocking_gates.py`, so defense-in-depth enforcement lives here directly.

Local diagnostic use of `--comparability-mode warn` or `off` at `bench/cli.py:58` is preserved; the trust boundary is the release entry points, not the dev CLI. Net effect: any compare-report JSON with `comparabilityPolicy.mode != "strict"` is rejected at the first release gate with a message naming the file, the observed mode, the required mode list, and the two ways to resolve it (re-run strict, or route through a non-release path). Per the (b) decision recorded in the prior audit: local exploration stays frictionless; release paths reject on read, every time.

No schema addition was required â€” `comparabilityPolicy.mode` is already serialized into compare artifacts today (verified against `bench/out/amd-vulkan/smoke/20260412T171500Z/compute_workgroup_atomic_1024.compare.json`, which carries `"comparabilityPolicy": {"mode": "strict", "requireTimingClass": "operation"}`).

**3. `allowBaselineNoExecution: true` justified at `compare.config.apple.metal.breadth.json`.**
The breadth config is the single compare config in `bench/native-compare/` that sets `allowBaselineNoExecution: true`; every other compare config in the directory sets it false. Reading the config: baseline runs `metal_doe_directional`, comparison runs `metal_dawn_release`, selector includes both `comparable` and `directional` benchmark classes, `claimability.mode: "off"`, `claimEligibleOnly: false`, mode is `warn`. This is a legitimate coverage-characterization lane, not a vestigial config â€” it intentionally exercises Doe-only coverage where Dawn may report `executionUnsupportedCount>0` / `executionSkippedCount>0`, which is exactly what `allowBaselineNoExecution` is designed for.

Justify-not-delete: added an inline `comparability.rationale` field to the config itself naming the lane's purpose, the two backend-lanes, the relaxation it grants, the guards that prevent promotion (`claimability.mode: "off"` + `claimEligibleOnly: false`), and the release-path rejection surface (`ensure_release_strict_comparability`). Rationale lives with the config so future readers don't need to trace back to status shards to understand why the flag is set.

### Why this matters

The three fixes address the combination called out in the prior audit: "a `--comparability-mode warn` run with an unknown-dispatch baseline, no claim gate, hits zero of the strict guards and emits a compare-report.json byte-identical in shape to a strict-mode comparable artifact." After this push:

- unknown-dispatch baselines fail the execution-shape parity check (fix #1),
- release paths reject `mode != strict` at read time (fix #2),
- and the single config that legitimately uses `allowBaselineNoExecution: true` carries its own rationale so it's auditable in isolation (fix #3).

## SPIR-V subgroup-op bug fixed; rmsnorm subgroup variant LOCAL-claimable on AMD Vulkan; second Doe-Vulkan pipeline bug surfaced as follow-up (2026-04-16 UTC)

### Bug #1 (fixed): Doe SPIR-V emitter passed Group Operation as `<id>` instead of literal

`runtime/zig/src/doe_wgsl/emit_spirv_builtins.zig::emit_subgroup_reduce`, `emit_subgroup_exclusive_scan`, and `emit_subgroup_inclusive_scan` all built an `OpConstant` for the SPIR-V GroupOperation enum (`Reduce`, `ExclusiveScan`, `InclusiveScan`) and then passed the resulting SSA `<id>` as the `Group Operation` operand of the `OpGroupNonUniform*` instruction. Per the SPIR-V grammar, `Group Operation` is a **literal** enum value, not an `<id>`. The result was technically-emittable SPIR-V where the literal slot held an arbitrary id number (e.g. `54`) instead of `0` (Reduce). `spirv-val` rejected it with `Invalid group operation operand: 54`, and RADV's pipeline compiler segfaulted in `vkCreateComputePipelines`.

- Fix: pass `spirv.GroupOperation.Reduce` / `ExclusiveScan` / `InclusiveScan` as a u32 literal directly into the operand list. No `const_u32` allocation, no SSA id involved.
- Verification: `spirv-val /tmp/rmsnorm_subgroup.spv` now passes silently. `spirv-dis` output now shows `OpGroupNonUniformFAdd %float %uint_3 Reduce %100` â€” `Reduce` rendered as a literal mnemonic, not an id reference.
- Tests: `zig build test`, `zig build test-wgsl`, `zig build bench-compilation -Doptimize=ReleaseFast` all exit 0 after the fix; existing test surface had no execution-path coverage for subgroup ops, so nothing broke.

This bug would have prevented **every** WGSL subgroup op (subgroupAdd, subgroupMin, subgroupMax, subgroupMul, subgroupAnd, subgroupOr, subgroupXor, subgroupExclusiveAdd, subgroupInclusiveAdd) from ever executing through Doe on a real Vulkan GPU. The compile-tests exercise textual MSL output and binary SPIR-V emission but never round-trip the SPIR-V through a real driver. This is a meaningful coverage gap that should be closed by adding a `spirv-val` step to CI for kernel artifact emission, plus a small "execute one subgroup kernel on Vulkan" smoke test in `runtime/zig/tests/` if/when a Vulkan-equipped CI host exists.

## WGSL compiler arena allocator: Doe now beats Tint warm in-process on every Unity production shader; prior Debug-vs-Release artifact retracted (2026-04-16 UTC)

Targeted at the apparent "Tint warm in-process is ~25Ă— faster than Doe on production Unity shaders" finding from the earlier 2026-04-16 entry below. Two compounding root causes were identified and fixed; the prior numbers were measuring a partly-broken comparison.

### Root cause 1: prior Doe binary was Debug, not Release

The earlier session built `runtime/zig/zig-out/bin/doe-compilation-bench` with bare `zig build bench-compilation`, which defaults to `-Doptimize=Debug`. The Tint-side binary (`bench/vendor/dawn/out/Release/{tint,tint_benchmark}`) is built with `is_debug = false` per the existing `args.gn`. The earlier "Doe `-95.9%` warm" reading on the Tint corpus was Doe-Debug vs Tint-Release. Rebuilding Doe with `zig build bench-compilation -Doptimize=ReleaseFast` (5.16 MB stripped binary vs 24.8 MB Debug) immediately moved the corpus warm delta from `-95.9%` to **`+116.5%`** (Doe ~2.16Ă— faster than Tint warm) before any code change. Doe Release on the empty-compute shader dropped from `~365Âµs` p50 to `~25Âµs` p50, a 14Ă— swing entirely attributable to the build mode.

### Root cause 2: per-compile allocator overhead inside `translateTo{Msl,Hlsl,Spirv,Dxil}`

Even after the build-mode fix, Doe carried an avoidable per-compile allocation tax. The pipeline `parser.parseSource -> sema.analyze -> ir_builder.build -> ir_validate.validate -> ir_transform_robustness.apply -> emit_*.emit` was running every internal allocation through whatever `std.mem.Allocator` the caller passed (typically `GeneralPurposeAllocator`). Tint's `tint_benchmark` warm samples don't pay that bookkeeping per iteration because Tint uses internal arenas/pools; the result was a fair ReleaseFast comparison still tilting in Tint's favour by a meaningful margin.

- Code (`runtime/zig/src/doe_wgsl/mod.zig`): wrapped each public translate entry point (`translateToMslWithOverrides`, `translateToHlslWithOverrides`, `translateToSpirv`, `translateToDxil`) in a `std.heap.ArenaAllocator` derived from the caller's allocator. The arena is passed to `analyzeToIr` so parser/sema/builder/validator/robustness all allocate into it. `module.deinit()` is removed (no-op against an arena), and the arena's own `deinit` releases everything in O(arena_count) rather than O(allocations) at end of translate. Emit calls (`emit_msl.emit`, `emit_hlsl.emit`, `emit_spirv.emit`, `emit_dxil.emit`) take only `(*const ir.Module, []u8)` and never allocate, so they are unaffected.
- Caller-visible contract: unchanged. The caller still passes any `std.mem.Allocator`. The arena lives entirely inside the translate call. Output bytes still land in the caller-supplied `out: []u8` buffer; nothing in the result references arena memory after return.
- Tests: `zig build test` (full unit suite) and `zig build test-wgsl` (WGSL compiler tests) both pass on Linux ReleaseSafe; the three pre-existing `warn` lines in test stderr are unchanged negative-path coverage for unsupported sType / WGSLâ†’MSL translation failure / Metal library null and are not regressions from this change.

## Root cause identified: Apple Metal package lanes are slow due to per-submit CPU-GPU serialization in doeNativeQueueSubmit (2026-04-08 UTC)

- finding:
  - `runtime/zig/src/doe_queue_submit_native.zig:204` calls `flush_pending_work(q)` at the
    start of every Metal `doeNativeQueueSubmit` call
  - `flush_pending_work` â†’ `flushPendingWorkTimed`
    (`runtime/zig/src/doe_queue_flush_breakdown.zig:45`) blocks on
    `metal_bridge_shared_event_wait` until the previous MTLCommandBuffer completes before
    encoding the next one
  - the gemma270m inference workload calls `queue.submit` 1583 times (413 writeBuffers +
    1170 dispatches); each call incurs one full CPU-GPU round-trip
  - measured cost per cycle: ~0.117ms; total accumulated: ~185.9ms
    (`submitAddonQueueSubmitTotalNs`) vs Dawn Metal's 3.5ms total (`submitQueueSubmitTotalNs`)
    for the same 1583 submits â€” a 53Ă— per-submit overhead gap
- why Vulkan does not have this:
  - `doeNativeQueueSubmit` returns immediately for the Vulkan backend (line 192â€“194); Vulkan
    commands are recorded synchronously during `dispatchWorkgroups`, so the submit path is a
    no-op with zero per-call overhead
- why this was not caught earlier:
  - the symptom (large `submitQueueSubmitTotalNs` / `submitAddonQueueSubmitTotalNs` on Apple
    package lanes) was tracked; the specific mechanism â€” `flush_pending_work` blocking on the
    shared event at submit-entry rather than deferring the wait â€” was not diagnosed
- related D3D12 note:
  - the current D3D12 queue-submit shim shows the same pre-submit CPU fence-wait
    anti-pattern: `submit_d3d12_commands` calls `rt.flush_queue()` before encoding
    a new command list
  - but D3D12 is broader than the old Metal bug: that shim also waits again after
    submit (`queue_execute` â†’ `queue_signal` â†’ `d3d12_bridge_fence_wait`), and the
    same synchronous submit-and-wait model appears in the current D3D12
    compute/upload runtime paths
  - so this should not be summarized as "same bug, same fix"; D3D12 needs a
    broader outstanding-fence/deferred-wait audit, and the current queue-submit
    shim only covers recorded render-pass submission
- fix target:
  - for Metal, remove or defer `flush_pending_work(q)` at the entry of
    `doeNativeQueueSubmit`; `MTLCommandQueue` guarantees sequential execution of
    committed command buffers without requiring CPU-side serialization between
    submits
  - the wait is only necessary before CPU reads (e.g. `mapAsync`), not before every new encode
