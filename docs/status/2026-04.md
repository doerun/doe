## Apple Metal warm package lanes improved again after deferring bound compute state until dispatch (2026-04-14 UTC)

- runtime/package changes:
  - the Node and Bun `doe-gpu` package backends now keep pending compute-pass
    pipeline and bind-group state in JS until dispatch instead of immediately
    crossing into native on every `setPipeline()` / `setBindGroup(0, ...)`
  - when the pending state is only the common `pipeline + bindGroup(0)` shape,
    dispatch now takes the existing native `computePassDispatchBound` fast path
    instead of paying three separate addon/FFI calls

    underlying WebGPU call, so the optimization stays narrow and semantics stay

## Root cause identified: Apple Metal package lanes are slow due to per-submit CPU-GPU serialization in doeNativeQueueSubmit (2026-04-08 UTC)

- finding:
  - `runtime/zig/src/doe_queue_submit_native.zig:204` calls `flush_pending_work(q)` at the
    start of every Metal `doeNativeQueueSubmit` call
  - `flush_pending_work` â†’ `flushPendingWorkTimed`
    (`runtime/zig/src/doe_queue_flush_breakdown.zig:45`) blocks on
    `metal_bridge_shared_event_wait` until the previous MTLCommandBuffer completes before
    encoding the next one
  - the gemma270m inference workload calls `queue.submit` 1583 times (413 writeBuffers +
    1170 dispatches); each call incurs one full CPU-GPU round-trip
  - measured cost per cycle: ~0.117ms; total accumulated: ~185.9ms
    (`submitAddonQueueSubmitTotalNs`) vs Dawn Metal's 3.5ms total (`submitQueueSubmitTotalNs`)
    for the same 1583 submits â€” a 53Ă— per-submit overhead gap
- why Vulkan does not have this:
  - `doeNativeQueueSubmit` returns immediately for the Vulkan backend (line 192â€“194); Vulkan
    commands are recorded synchronously during `dispatchWorkgroups`, so the submit path is a
    no-op with zero per-call overhead
- why this was not caught earlier:
  - the symptom (large `submitQueueSubmitTotalNs` / `submitAddonQueueSubmitTotalNs` on Apple
    package lanes) was tracked; the specific mechanism â€” `flush_pending_work` blocking on the
    shared event at submit-entry rather than deferring the wait â€” was not diagnosed
- related D3D12 note:
  - the current D3D12 queue-submit shim shows the same pre-submit CPU fence-wait
    anti-pattern: `submit_d3d12_commands` calls `rt.flush_queue()` before encoding
    a new command list
  - but D3D12 is broader than the old Metal bug: that shim also waits again after
    submit (`queue_execute` â†’ `queue_signal` â†’ `d3d12_bridge_fence_wait`), and the
    same synchronous submit-and-wait model appears in the current D3D12
    compute/upload runtime paths
  - so this should not be summarized as "same bug, same fix"; D3D12 needs a
    broader outstanding-fence/deferred-wait audit, and the current queue-submit
    shim only covers recorded render-pass submission
- fix target:
  - for Metal, remove or defer `flush_pending_work(q)` at the entry of
    `doeNativeQueueSubmit`; `MTLCommandQueue` guarantees sequential execution of
    committed command buffers without requiring CPU-side serialization between
    submits
  - the wait is only necessary before CPU reads (e.g. `mapAsync`), not before every new encode
- follow-up:
  - implement deferred-flush behavior for the Metal NAPI path and verify with a fresh
    gemma270m Metal node-package and bun-package compare run
  - confirm `submitAddonQueueSubmitTotalNs` drops to match Vulkan-range overhead after fix
  - audit the D3D12 runtime-wide fence/wait model separately instead of assuming
    the Metal queue patch generalizes unchanged
