## SPIR-V graphics path and robustness verified complete; Metal native smoke green on all 3 workload types (2026-04-15 UTC)

Strategy-doc gap assessment for "SPIR-V graphics path completion" and "robustness transform atomic bounds" was investigated and found to be stale — both features are already implemented and tested.

- **SPIR-V graphics stages**: `runtime/zig/src/doe_wgsl/emit_spirv_stages.zig` already handles struct-typed vertex inputs (decomposed into per-field Input variables via `emit_struct_io_vars`, lines 166-186; loaded and CompositeConstructed in the wrapper body, lines 130-140), MRT fragment outputs (per-field Output variables via `emit_return_output_vars`, lines 197-209; decomposed via CompositeExtract in `store_return_to_outputs`, lines 262-273), texture/sampler promotion (audit at lines 33-51 confirms correctness), and all IO decorations (Location, BuiltIn, interpolation, sampling, invariant, blend_src at lines 284-321). Tests in `tests/wgsl/emit_spirv_stage_test.zig` exercise: vertex entry point emission, fragment entry point emission, struct return with position+location, struct input parameter with 2 locations, and MRT fragment output with 2 color targets. `zig build test-wgsl` exit 0.
- **Robustness atomic bounds**: `ir_transform_robustness.zig` already clamps all array index operations (lines 61-93), including indices into `array<atomic<T>>`. Since atomic builtin calls (atomicLoad, atomicStore, etc.) take a pointer derived from an already-clamped array element reference, atomic access is transitively bounded. Verified by compiling a `counters: array<atomic<u32>>` workload with `atomicAdd(&counters[gid.x], 1u)` — compiled successfully to MSL (558µs) and SPIR-V, confirming the robustness pass ran on the index expression before the atomic pointer was formed.
- **Metal native backend smoke** (`apple-metal-backend-smoke`, 3 workloads): compute_workgroup_atomic_1024, render_draw_throughput_baseline, and upload_write_buffer_64kb all ran successfully on both Doe and Dawn sides. Doe faster on all 3: compute +15% (47ms vs 55ms), render +93% (0.5ms vs 7.3ms), upload +99.7% (0.0004ms vs 0.11ms). Results are diagnostic with only 2 samples and 0 warmup; the render and upload deltas need more samples per apples-to-apples discipline. Key evidence: render pipeline (vertex + fragment stages, draw commands, render pass recording) works end-to-end on Metal via Doe native runtime.
- **Metal native compare-dev** (25 workloads, `apple-metal-backend-compare-dev`): 25/25 pass on both Doe and Dawn. Doe faster on 23/25. Two render regressions at high draw counts: `render_bundle_dynamic_pipeline_bindings_200k` (Doe 121ms vs Dawn 99ms, -22.7%) and `render_draw_throughput_200k` (Doe 4.7ms vs Dawn 3.4ms, -36.7%). Root cause: structural per-draw ObjC message dispatch overhead in Doe's direct Metal API pattern (`metal_bridge_render_encoder_draw` at `metal_bridge.m:942-951`) vs Dawn's batched C++ command recording layer. The draw loop itself is tight (conditional pipeline bind + `[encoder drawPrimitives:]` per draw); the delta is inherent to the bridging approach. Not a bug; a known architectural tradeoff.
  - Doe wins include: pipeline_compile_stress +50%, texture_sampler_write_query_destroy +85%, upload_write_buffer_1mb_staged +99.6%, compute_zero_initialize_workgroup_memory_256 +70%, render_draw_redundant_pipeline_bindings_200k +69%, render_draw_state_bindings +62%, upload_write_buffer_256mb_staged +44%
  - All results are diagnostic (2 samples, 0 warmup per config); not claimable per apples-to-apples discipline
  - follow-up: consider command batching layer for high-draw-count render workloads if the delta needs to be closed; currently acceptable for all other workload types

## Root cause identified: Apple Metal package lanes are slow due to per-submit CPU-GPU serialization in doeNativeQueueSubmit (2026-04-08 UTC)

- finding:
  - `runtime/zig/src/doe_queue_submit_native.zig:204` calls `flush_pending_work(q)` at the
    start of every Metal `doeNativeQueueSubmit` call
  - `flush_pending_work` → `flushPendingWorkTimed`
    (`runtime/zig/src/doe_queue_flush_breakdown.zig:45`) blocks on
    `metal_bridge_shared_event_wait` until the previous MTLCommandBuffer completes before
    encoding the next one
  - the gemma270m inference workload calls `queue.submit` 1583 times (413 writeBuffers +
    1170 dispatches); each call incurs one full CPU-GPU round-trip
  - measured cost per cycle: ~0.117ms; total accumulated: ~185.9ms
    (`submitAddonQueueSubmitTotalNs`) vs Dawn Metal's 3.5ms total (`submitQueueSubmitTotalNs`)
    for the same 1583 submits — a 53× per-submit overhead gap
- why Vulkan does not have this:
  - `doeNativeQueueSubmit` returns immediately for the Vulkan backend (line 192–194); Vulkan
    commands are recorded synchronously during `dispatchWorkgroups`, so the submit path is a
    no-op with zero per-call overhead
- why this was not caught earlier:
  - the symptom (large `submitQueueSubmitTotalNs` / `submitAddonQueueSubmitTotalNs` on Apple
    package lanes) was tracked; the specific mechanism — `flush_pending_work` blocking on the
    shared event at submit-entry rather than deferring the wait — was not diagnosed
- related D3D12 note:
  - the current D3D12 queue-submit shim shows the same pre-submit CPU fence-wait
    anti-pattern: `submit_d3d12_commands` calls `rt.flush_queue()` before encoding
    a new command list
  - but D3D12 is broader than the old Metal bug: that shim also waits again after
    submit (`queue_execute` → `queue_signal` → `d3d12_bridge_fence_wait`), and the
    same synchronous submit-and-wait model appears in the current D3D12
    compute/upload runtime paths
  - so this should not be summarized as "same bug, same fix"; D3D12 needs a
    broader outstanding-fence/deferred-wait audit, and the current queue-submit
    shim only covers recorded render-pass submission
- fix target:
  - for Metal, remove or defer `flush_pending_work(q)` at the entry of
    `doeNativeQueueSubmit`; `MTLCommandQueue` guarantees sequential execution of
    committed command buffers without requiring CPU-side serialization between
    submits
  - the wait is only necessary before CPU reads (e.g. `mapAsync`), not before every new encode
- follow-up:
  - implement deferred-flush behavior for the Metal NAPI path and verify with a fresh
    gemma270m Metal node-package and bun-package compare run
  - confirm `submitAddonQueueSubmitTotalNs` drops to match Vulkan-range overhead after fix
  - audit the D3D12 runtime-wide fence/wait model separately instead of assuming
    the Metal queue patch generalizes unchanged

## 2026-04-23

- TSIR Step 4 — thirty-ninth increment: `.fused_gemv`
  promotion now uses structural matrix-access evidence instead
  of the temporary same-bound exclusion from increment 37. The
  family classifier moved from `frontend.zig` into
  `runtime/zig/src/tsir/family_hint.zig`; the frontend now
  delegates to that shard. A reduction promotes to
  `.fused_gemv` only when it has exactly 2 axes, one sum
  reduction on axis 1, a non-trivial outer bound, and an
  indexed global load whose expression tree references both
  the outer and inner axis locals (for example `W[i * K + k]`
  or `W[i][k]`). RMSNorm no longer depends on the fragile
  "same upper bound" guard: its split accesses (`input[i]`,
  then `input[d]` / `weight[d]`) do not satisfy the matrix
  access predicate, so it stays on `.reduction` until a real
  `.rms_norm` detector lands. The focused GEMV unit now uses
  the actual matrix access pattern, and `zig build test-wgsl`
  passes.
- TSIR Step 4 — thirty-eighth increment: dispatch-axis bound
  detection now scans past interleaved `let` decls and
  early-return guards for OTHER dispatch axes to find the
  matching guard. New helper `scanForDispatchGuard(allocator,
  module, function, body_range, start_pos, dispatch_local)`
  in `runtime/zig/src/tsir/frontend.zig` walks forward in a
  block starting from the position after a dispatch local's
  decl; skips through subsequent local_decls; at each if it
  tries `extractDispatchBoundFromGuard` and returns on match;
  if the if is a bare early-return (no else, then-body is a
  single-return) for some OTHER local, skips past and keeps
  scanning; anything else stops the scan (arithmetic assigns,
  loops, non-guard ifs) and the caller falls back to the
  placeholder. Why the skip-safe predicate matters: a
  multi-axis dispatch kernel like gather has
  `let t; let h; if (t >= ...) return; if (h >= ...) return;`
  where the first decl's matching guard is TWO positions
  ahead, and the second decl's matching guard is also two
  positions ahead past an unrelated guard. The scan handles
  both. Updated the gather bootstrap integration test to
  assert the real resolved bounds:
  `"uniform:u.num_tokens"` for the `t` axis and
  `"uniform:u.hidden"` for the `h` axis. All three Phase A
  bootstrap kernels now lower to TSIR semantic with real
  bounds — no placeholder strings, no spurious rejections.
  With this, Step 4's frontend is green for Phase A purposes:
  axes, bindings, reductions, collectives, family hints, and
  typed rejections all populate correctly for the bootstrap
  catalog. Remaining Step 4 plan bullets (SSA normalization,
  full access-stride summary analysis) are deferred as
  later-phase hardening; they don't block Step 5 residency
  planning for the bootstrap kernels. `zig build test-wgsl`
  reports 901/901 tests passed.
- TSIR Step 4 — thirty-seventh increment: two-part close-out
  for the frontend pass.
  (1) Tightened the `.fused_gemv` heuristic in
  `runtime/zig/src/tsir/frontend.zig` to require
  `axes[0].upper_bound != axes[1].upper_bound`. The previous
  rule over-matched: any 2-axis kernel with a reduction on
  axis 1 and a non-trivial outer bound got tagged
  `.fused_gemv`. RMSNorm has the same shape but BOTH axes are
  bounded by the same `u.hidden_size`, which the tightened
  rule now rejects as a GEMV candidate — it stays on
  `.reduction`. The catch: when the planner eventually lands
  `.rms_norm` / `.layer_norm` detection, the distinguishing
  signal (same-upper-bound pattern + post-reduction normalize
  step) gets added as a positive refinement; the current
  coarse fallback is the honest intermediate.
  (2) Added three bootstrap integration tests that load
  `runtime/zig/tests/tsir/bootstrap/{fused_gemv,rms_norm,gather}.wgsl`

  for each. fused_gemv: 2 axes `[i, k]` with bounds
  `uniform:u.M` / `uniform:u.K`, one sum reduction on axis 1,
  hint `.fused_gemv`, zero rejections. rms_norm: 2 axes `[d,
  i]` with bounds both `uniform:u.hidden_size`, one sum
  reduction on axis 1, hint `.reduction` (coarse fallback
  because both upper bounds match). gather: 2 axes `[t, h]`
  with placeholder upper bounds `dispatch.y` / `dispatch.x`
  (the guards aren't adjacent to each let; a later

  the canonical inputs lower end-to-end through the frontend
  and produce stable, auditable TSIR. `zig build test-wgsl`
  reports 901/901 tests passed (up from 898).
- TSIR Step 4 — thirty-sixth increment: family hint now
  refines `.reduction` → `.fused_gemv` for the canonical
  two-axis shape: exactly 2 axes, 1 reduction on axis 1
  (inner), and the outer axis's upper bound is non-trivial
  (dispatch / uniform / override / const prefix). New
  predicate `isNontrivialBoundString(s)` in
  `runtime/zig/src/tsir/frontend.zig` matches the leading
  prefix set so `"dispatch.x"`, `"uniform:u.M"`,
  `"override:N"`, `"override@id:5"`, and `"const:K"` all
  qualify, while fixed literals like `"4"` stay coarse.
  Guard rationale: a kernel with fixed literal bounds is
  technically a matmul, but its compile-time-constant outer
  extent doesn't benefit from the GEMV-specific planner
  strategy (streaming weights, row-splitting), so the
  coarse hint is more honest. New tests: "frontend refines
  reduction hint to fused_gemv for canonical 2-axis shape"
  drives the full fused-GEMV WGSL (dispatch-grid outer,
  uniform-bounded inner) and asserts
  `family_hint == .fused_gemv`; "frontend keeps reduction
  hint for fixed-literal outer bound" drives the same

  `.reduction`. `zig build test-wgsl` reports 898/898 tests
  passed.
- TSIR Step 4 — thirty-fifth increment: family-hint inference
  now distinguishes `.gather` from `.elementwise`. New helpers
  `hasIndirectBufferAccess(function)` and
  `indexExprContainsBufferIndex(function, expr_id)` in
  `runtime/zig/src/tsir/frontend.zig` walk `function.exprs`
  looking for any `.index` expression whose index-field
  expression tree contains another `.index` node. Matching
  that shape — the canonical data-dependent buffer load
  `input[indices[i]]` — promotes the family hint from
  `.elementwise` to `.gather`. `inferFamilyHint`'s signature
  now also takes the function pointer so the expression walk
  is reachable. Critical no-false-positive property: linear
  index combinations that happen to include uniform scalar
  loads (`W[i * K + k]` where `K` is a uniform) do NOT
