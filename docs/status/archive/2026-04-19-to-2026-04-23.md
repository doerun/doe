## Push: WGSL postfix ++/-- support + emit_spirv error surfacing (2026-04-23 UTC)

**Landed:** Doe's WGSL compiler now supports postfix increment and decrement
statements (`x++`, `x--`), which are standard WGSL per the W3C spec and are
used extensively across Doppler's production compute kernels. Error surfacing
through the translation pipeline is tightened so downstream failures carry
detail instead of an empty "WGSL→SPIR-V translation failed:" message.

- `runtime/zig/src/doe_wgsl/token.zig`: new `plus_plus` / `minus_minus`
  token tags.
- `runtime/zig/src/doe_wgsl/lexer.zig`: two-char lookahead emits the new
  tokens for `++` and `--`.
- `runtime/zig/src/doe_wgsl/ast.zig`: new `inc_stmt` / `dec_stmt` node tags.
- `runtime/zig/src/doe_wgsl/parser_stmt.zig`: `parseExprOrAssignStmt` now
  recognises `lhs ++` / `lhs --` at statement position.
- `runtime/zig/src/doe_wgsl/sema.zig` and `sema_body.zig`: analyze the new
  statements as ref-returning lvalue access with the lhs type.
- `runtime/zig/src/doe_wgsl/ir_builder.zig`: lower `.inc_stmt` / `.dec_stmt`
  to `{ assign, op = .add or .sub, lhs, rhs = int_lit(1) }`, synthesising
  a `1` expression at IR time with `semantic.abstract_int_type`.
- `runtime/zig/src/doe_wgsl/mod.zig`: surface ir_transform / ir_opt_rewrite
  errors through `setLastErrorDetail` instead of swallowing them as `OOM`;
  new `ir_transform` compilation stage; `setLastErrorDetailPublic` exported
  for `runtime_compile` callers.
- `runtime/zig/src/doe_wgsl/runtime_compile.zig`:
  `translateToSpirvForComputeRuntime` now records emit_spirv error kind via
  `setLastErrorDetailPublic` instead of returning silently.
- `runtime/zig/src/doe_vulkan_compute_native.zig`: shader-compile error log
  now includes the first 120 chars of the failing WGSL so kernels can be
  identified from stderr alone.

**Current evidence:** `zig build test-wgsl` exit 0. All 21 related Python
unit tests pass. Running `bench/tools/run_doe_webgpu_shared_contract.py`
against the shared execution contract now gets past the prior
`sema: UnknownIdentifier at 357:47 near "for (var s = 0u; s < num_subgroups; s++)"`
failure on `rmsnorm.wgsl`. Three production compute kernels still fail,
but now with specific, identifiable `emit_spirv` errors:

- `Q4_K Dequantization Kernel - f16 Output (vec4)` — `spirv_emit: InvalidIr`.
- `RMSNorm Kernel with Fused Residual Add` — `spirv_emit: InvalidIr`.
- `Fused Multi-Head Attention Kernel (head_dim 256, f16 KV)` —
  `spirv_emit: UnsupportedConstruct`.

**Blocker:** each remaining failure is a distinct emit_spirv gap, likely in
`unpack2x16float`, `override` constant handling, f16 vector lanes, or
workgroup-barrier-heavy attention patterns. These are separate kernel-level
fixes in `runtime/zig/src/doe_wgsl/emit_spirv*.zig`, not a single pass.
Producing a green reference-vs-WebGPU parity receipt is gated on these three
kernels compiling cleanly.

**Follow-up:** next session picks up the three kernel gaps one at a time.
The WGSL snippet in stderr identifies each failing kernel; the
`spirv_emit: InvalidIr` / `UnsupportedConstruct` tag points at the
category of fix.

  params.
- `bench/tests/test_int4ple_scheduler_readiness.py` locks the all-projected,
  held-diagnostic, and mixed-projected/unprojected code paths.

**Current evidence on this host (SDK container reachable via
`/home/x/cerebras-sdk/cslc` and `.direct-rootfs`):** the fresh
`simulator-driver-result.json` reports 10 of 14 targets compile cleanly at
manifest scale: `residual`, `rmsnorm`, `gelu`, `sample`, `rope`, `tiled`,
`gemv`, `lm_head_prefill_stable`, `attn_decode`, `final_norm_stable`. The
remaining 4 fail compile with real kernel-level errors that the manifest
promotion surfaces honestly:

- `embed`: `integer value 49152 cannot be coerced to type 'i16'` at
  `compile/embed/pe_program.csl:15`. Root cause: the kernel allocates
  `[num_tokens * hidden_size]f32` per PE. CSL array dims are i16 (max 32767)
  and PE memory is ~48KB; at projected manifest `num_tokens=max_seq_len=4096,
  hidden_size=1536` the per-PE output alone is ~24MB. The full
  `[rows_per_pe * hidden_size]f32` table is also over PE memory at 96KB.
  Kernel requires architectural redesign: quantized table storage
  (q4k, ~0.5 bytes/element + scales), dequant-on-gather, and likely fabric
  routing for output assembly.
- `lm_head_gemv_stable`: `integer value 383328 cannot be coerced to type
  'i16'` at `compile/lm_head_gemv_stable/pe_program.csl:23`. Root cause: the
  kernel allocates `[out_dim * num_blocks_per_row * 144]u8` per PE =
  `1331 * 2 * 144 = 383KB`. Same class as embed: weight table per PE exceeds
  i16 dim and PE memory budget. Kernel requires architectural redesign.
- `attn_head256`: `ld.lld: error: ran out of PE memory for data
  (section .bss)`, `.bss` range 0x1414..0x25513 (≈145KB) overlaps `.filters`.
  Root cause: `emit_csl_attention.zig` allocates full
  `K[kv_len * head_dim]f32`, `V[kv_len * head_dim]f32`, `Q[q_len * head_dim]f32`,
  `O[q_len * head_dim]f32` backing arrays per PE. At `kv_len=q_len=24,
  head_dim=256` that is already 96KB of scratch.
- `attn_head512`: same class, larger — `.bss` ≈295KB.

**Not an infrastructure gap:** the SDK is reachable, `cslc` runs, and
`ld.lld` is emitting real linker diagnostics. Earlier
`csl_compile_container_runtime_blocked` classifications in older receipts were
stale; the fresh run reaches actual kernel compilation and surfaces these
four architectural kernel gaps explicitly.

**Blocker:** Full CSL simulator end-to-end for Gemma 4 E2B requires
architectural redesign of these four kernel emitters (quantized gather,
streamed fused-gemv, tiled attention with bounded per-PE scratch). This is
real kernel engineering work and is not in-session scope. A separate
long-running thread should pick these up one at a time
(embed → lm_head_gemv_stable → attn_head256 → attn_head512).

**Follow-up:** Projection rules for `rmsnorm`, `residual`, `gelu`, and
`final_norm_stable` are unnecessary — those kernels have no shape-dependent
compile params and compile cleanly at manifest fabric dims. Projection rule
for `lm_head_prefill_stable` remains open (SUMMA shape for
prompt-length × hidden × vocab still to be decided). Python sharding debt on
`bench/tools/run_doe_csl_int4ple_transcript.py` (over 1200 lines) remains
active.
